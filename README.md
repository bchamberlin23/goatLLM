# goatLLM

goatLLM is a desktop AI workspace for chat, coding agents, design artifacts, research, notebooks, memory, and project-aware automation. It is built with Tauri 2, React 19, Vite, Zustand, Rust commands, and the Vercel AI SDK.

The app is model-agnostic: it can use bundled free models, hosted API providers, OpenAI-compatible endpoints, LM Studio, and Ollama. The main design goal is a local-first assistant that can talk, inspect a workspace, make approved changes, generate artifacts, and survive abrupt app exits without losing user-visible work.

## What It Does

### Chat

Chat mode is the normal conversation surface. It supports streaming responses, markdown rendering, syntax-highlighted code, tables, math, attachments, editable messages, conversation search, exports, branches/forks, custom per-conversation prompts, context compaction, and collapsible reasoning blocks for models that emit thinking content.

Chat can also use selected lightweight tools when enabled:

- `web_search` and `scrape_url` for current information.
- `read_attachment` and `search_attachment` for uploaded documents.
- `load_skill` for progressive skill loading.
- `manage_memory` for long-term facts, preferences, contacts, and tasks.
- Optional approved scratch execution with `run_python` and `run_javascript`.

### Agent

Agent mode adds workspace-scoped tools. The agent can inspect code, search files, run tests, edit files, manage git, use browser fetch/extract tools, index workspaces for semantic search, and delegate independent subtasks to subagents.

Read-oriented tools include:

- `read_file`
- `list_dir`
- `search_content`
- `search_semantic`
- `git_status`
- `git_log`
- `git_blame`
- `read_pdf`
- attachment reading and searching
- web search and URL scraping
- `load_skill`

Write or side-effectful tools are approval-gated:

- `write_file`
- `edit_file`
- `bash` / `exec_command`
- `diff_file`
- `git_branch`
- `git_commit`
- `git_push`
- `run_tests`
- `read_lints`
- `browser_fetch`
- `browser_extract`
- `browser_session_open`
- `browser_session_navigate`
- `browser_session_close`
- `index_workspace`
- `run_python`
- `run_javascript`
- `manage_memory`

Agent prompts are generated by `src/lib/system-prompt.ts`. The prompt injects project instruction files such as `AGENTS.md`, `GOAT.md`, and `CLAUDE.md`, explains tool routing, encourages focused verification, and tells the agent to finish with `done`.

### Plan And Research Tool Sets

The tool registry exposes narrower bundles in addition to the full agent tool set:

- `PLAN_TOOLS` gives the model read-only workspace inspection plus todos.
- `RESEARCH_TOOLS` gives chat-mode research access to search, scraping, browser fetch/extract, attachments, skills, and memory.
- `CHAT_TOOLS` gives lightweight web and attachment navigation in normal chat.
- `ATTACHMENT_TOOLS`, `SKILL_TOOLS`, and `CODE_EXEC_TOOLS` are composed as needed by the input pipeline.

### Design And Artifacts

Design mode is for generating and refining side-panel artifacts. It combines design-system context, skill templates, discovery forms, critique support, and artifact extraction.

The shared artifact panel supports:

- HTML previews in a sandboxed iframe.
- LaTeX source with PDF compilation.
- Python artifacts with runnable output.
- Word, PowerPoint, and Excel generation from markdown/table sources.
- Monaco-backed source editing.
- Copy, download, print, preview/code toggles, and version-aware targeted edits through `edit_artifact`.
- Workspace file browsing and side-by-side file editing.

### Notebook

Notebook mode is feature-flagged and exposed as a fourth mode when enabled. It is a freeform board for text and runnable code cells with AI assistance. Notebook data is sanitized on hydrate so stale runtime-only state does not leak into restored sessions.

### Product Workspace Panel

`src/lib/product-workspace.ts` powers the product workspace primitives used by dashboard-style features:

- Usage and cost estimation by provider/model.
- Conversation branch graph summaries.
- Prompt document filtering.
- Notebook cell normalization.
- Model comparison runs.
- Image generation jobs.
- Scheduled agents.
- Watcher event summaries.
- Sync configuration normalization.

Runtime-only statuses are settled on hydrate. Interrupted model comparisons, image jobs, and scheduled agent runs are restored as errors instead of stale queued/running UI.

### Skills, Memory, Todos, MCP, And Subagents

goatLLM includes several agent productivity systems:

- Skills can be loaded on demand and auto-triggered by intent.
- Memory stores durable facts, contacts, preferences, and tasks.
- Todos persist across multi-step agent work.
- Subagents can be spawned for independent work and reviewed through transcript panels.
- MCP servers are supported through HTTP and stdio transports.

## Persistence

Messages and conversations use a dual-write strategy:

1. A synchronous localStorage journal under `goatllm-*` keys.
2. An async SQLite mirror through Tauri IPC and `@tauri-apps/plugin-store`.

Startup reads the journal first for instant recovery, then merges SQLite. The UX guarantee is that if a user sends a message and closes the app immediately, the message should still be present after restart.

New persistent features should follow the same rule: write user-visible durable data to both layers, reset runtime-only state on hydrate, and test the reload cycle.

Main source: `src/lib/db.ts`.

## Safety Model

goatLLM is intentionally conservative around local workspaces:

- Workspace paths are normalized and checked against the selected root.
- Sensitive paths and patterns such as `.env`, private keys, `.ssh`, and `secrets/` are denied.
- Write tools require approval unless the user changes permission mode.
- Shell commands are classified by risk.
- Dangerous git operations, destructive commands, and external effects are gated.
- Browser fetch tools block non-HTTP schemes and cloud metadata targets and cap output size.
- Artifact previews are sandboxed.
- The working tree may contain user changes; agents are instructed not to overwrite or discard unrelated work.

## Providers And Models

The provider layer supports cloud providers, local providers, and OpenAI-compatible endpoints. Built-in free models are seeded in `src/lib/providers.ts`:

- DeepSeek V4 Flash (Free)
- MiMo V2.5 (Free)
- Nemotron 3 Super (Free)

Settings manage API keys, custom base URLs, enabled models, local model options, semantic indexing settings, skills, MCP servers, and safety preferences.

## Getting Started

### Prerequisites

- Node.js 20+
- pnpm
- Rust stable
- Platform dependencies for Tauri 2
- macOS 12+ for the bundled macOS app target

### Install

```bash
git clone https://github.com/bchamberlin23/goatLLM.git
cd goatLLM
pnpm install
```

### Run The Desktop App

```bash
pnpm tauri dev
```

The Tauri dev server uses Vite at `http://localhost:1420`.

### Run The Web Frontend Only

```bash
pnpm dev
```

This is useful for fast UI iteration, though desktop-only Tauri commands will not work outside the app shell.

## Design System

`DESIGN.md` is the source of truth for visual work. It defines the Geist and JetBrains Mono font choices, the single amber accent (`#f59e42`), dark surface tokens, text contrast floors, spacing, radius, motion, and anti-patterns to avoid.

Read `DESIGN.md` before making visual or UI changes.

## Development Notes

- Keep user-visible durable data in both the localStorage journal and SQLite mirror.
- Reset in-progress UI state during hydrate.
- Prefer structured tool APIs over shell fallbacks in agent prompts and tests.
- Use focused regression tests for prompt, persistence, tool, and hydrate behavior.
- Do not add AI or bot co-author trailers to commit messages.
- Avoid force-pushes and destructive git operations unless explicitly requested.
